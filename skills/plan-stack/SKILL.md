Use this skill before building a project that has both UI and backend behavior.

## Goal

Create a simple project structure that a beginner can follow without using unnecessary architecture.

## Workflow

### Step 1

Understand the product in plain terms:

- what the user can do
- what pages exist
- what data needs to be stored
- what actions need backend support

### Step 2

Produce:

## Frontend Plan

- pages or screens
- main components
- key user actions
- state or data needed on each page

## Backend Plan

- routes or endpoints
- models or tables
- validation needs
- services or logic

## Shared Contracts

- request data
- response data
- important field names

## Suggested Folder Structure

- frontend folders
- backend folders
- shared folders if needed

## Build Order

- what to build first
- what to mock first
- what can wait

### Step 3

Keep the design simple unless the project truly needs more layers.

## Rules

- Avoid patterns that only make sense in large production systems.
- Prefer a structure a beginner can recognize quickly.
- Use the smallest number of moving parts that still makes sense.
- If frontend-only is enough for the first version, say so.
