Use this skill when the user shares a project brief, design brief, app idea, assignment, or feature description and wants to understand what to build.

## Goal

Turn the brief into something a beginner can act on without feeling overwhelmed.

## Workflow

### Step 1

Read the brief carefully and identify:

- the product goal
- the target user
- the main pages or screens
- the key features
- the most important actions the user should be able to take

### Step 2

Rewrite the work in simple language. Prefer plain words over technical jargon.

### Step 3

Produce:

## Brief Breakdown

- What this project is
- Who it is for
- Main pages or screens
- Main features
- Data the app will likely need
- Suggested build order

## Starter MVP

- What to include in version 1
- What screen or feature to build first
- What to leave for version 2

## First Small Steps

- Step 1
- Step 2
- Step 3

### Step 4

If the brief is too large for a beginner, say so clearly and reduce it to a smaller first version.

## Rules

- Keep the output short and practical.
- Do not jump straight into code unless the user asks.
- If the brief includes many screens, suggest the easiest one to build first.
- When helpful, recommend a simple MVP instead of the full scope.
- Prefer a version-1 plan the user could realistically finish.
